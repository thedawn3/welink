# WeLink AI 接手规范

把本文视为当前仓库默认的 AI 协作规范。

默认目标：

- 先理解项目，再决定动作
- 先兼容现有体系，再考虑补全
- 先做最小破坏，再考虑结构性治理

## 1. 启动方式

- 先做非侵入式探索，不要一上来就改文件
- 修改 Git 仓库前，先做远端一致性检查；优先使用 `$git-remote-sync-guard`
- 优先识别：
  - 主入口文档
  - 文档索引
  - 项目本地上下文
  - 关键代码目录
  - 运行入口、构建命令、测试命令、配置文件
- 能从仓库里发现的事实，不要先问用户

## 2. 现有资产兼容原则

- 根 `README.md` 是产品入口
- `docs/README.md` 是文档索引
- `docs/PROJECT_LOCAL_CONTEXT.md` 是项目特有规则真相源
- `docs/AI_PROJECT_STARTER/` 只承载跨项目可复用的 AI 协作经验
- 不要因为引入 starter，就再做一套并行文档入口

## 3. 文档治理原则

- 规则与展开说明分层
- 同一事实只保留一个主承载文档
- 如果新增内容已能归入现有章节，不要补丁式平行追加
- 只要改动影响流程、依赖、状态机、接口语义、联调方式，就要判断文档是否需要同步
- 如果改动影响多个入口，优先修入口再修下游文档

## 4. WeLink 联动意识

在 WeLink 里，下列变化通常不是单点改动：

- 启动 / 路径 / compose / `.env`
- 索引流程
- MCP 接入说明
- 关系分析口径
- 新增后端接口
- 联系人详情入口与展示

遇到这些变化时，要主动检查相关文档、类型、前端 API 层、Swagger 是否同步。

## 5. 执行边界

- 未经用户要求，不要删除或重写整套文档体系
- 若远端一致性检查异常，先告知用户，不要继续编辑
- 不要自动执行 `pull`、`rebase`、`merge`
- 如果当前仓库未发现某类审计/同步脚本，默认记为“未发现，已跳过”，不要把模板经验当成本仓库事实
